# J2 — HTML & Accessibility: Semantic Markup Audit

**Level:** Junior
**Skills tested:** Semantic HTML5, ARIA roles/attributes, form accessibility, keyboard navigation
**Files to edit:** `src/index.html`, `src/app.component.html`, `src/restaurant/reservation/reservation.component.html`

---

## Context

The app has three accessibility problems:
1. The toolbar navigation buttons give no indication of which view is currently active — screen readers and keyboard users can't tell where they are.
2. The reservation form's error messages are not linked to their inputs — a screen reader announces the error text, but the user doesn't know which field it belongs to.
3. There is no way to skip the toolbar and jump directly to the main content via keyboard.

Your task is to fix all three without altering any TypeScript logic or visual design.

## Tasks

### Task 1 — Skip navigation link
Add a visually-hidden "Skip to main content" anchor at the very top of `index.html`. It should:
- Be invisible by default (use CSS `position: absolute; left: -9999px` or the `.sr-only` pattern)
- Become visible when focused (`:focus` state)
- Link to `#main-content`

Also add `id="main-content"` to the `<div class="content-container">` in `app.component.html`.

### Task 2 — Active state on nav buttons
The toolbar buttons have no `aria-pressed` attribute. Add it so that:
- The active button has `[attr.aria-pressed]="true"`
- The inactive button has `[attr.aria-pressed]="false"`

Bind these to the existing `showMenu` and `showReservation` boolean flags already in `AppComponent`.

### Task 3 — Form error message association
Each `<mat-error>` block renders an error but is not programmatically linked to its input. For each form field:
1. Add a unique `id` to the error element (e.g., `id="name-error"`)
2. Add `aria-describedby="name-error"` to the corresponding `<input>`
3. Add `aria-invalid` bound to the control's invalid + touched state

---

## Acceptance criteria

- [ ] "Skip to main content" link appears on keyboard focus at the top of the page
- [ ] `aria-pressed="true"` is present on the currently active nav button
- [ ] `aria-pressed="false"` is present on inactive nav buttons
- [ ] Each form input has `aria-describedby` pointing to its error element's `id`
- [ ] Each form input has `aria-invalid="true"` when the field is invalid and touched
- [ ] No TypeScript changes beyond binding attributes in the template
