# A3 — Code Review & Architectural Critique

**Level:** Architect
**Skills tested:** Code quality judgment, identifying real bugs, security awareness, error boundaries, API design, accessibility, coupling
**Files involved:** All existing source files (read-only audit first, then targeted fixes)

---

## Context

This challenge has no single "correct" solution. The candidate reviews the full codebase, identifies architectural and quality issues, prioritises them by impact, and implements fixes for the three most critical ones.

A strong architect doesn't just list problems — they explain *why* it's a problem, what could go wrong in production, and how the fix improves the system.

---

## Phase 1 — Audit (20 minutes)

Read these files in order and note every issue you find:

1. `src/restaurant/services/order-tracking.service.ts`
2. `src/restaurant/order-tracker/order-tracker.component.ts`
3. `src/restaurant/reservation/reservation.component.ts`
4. `src/restaurant/menu/menu.component.ts`
5. `src/app.component.ts` + `src/app.component.html`

For each issue, classify it:
- **Bug** — causes incorrect behaviour right now
- **Memory / resource leak** — causes degraded performance over time
- **Coupling** — makes the code hard to change
- **Missing error handling** — causes silent failures
- **Accessibility** — excludes users
- **API design smell** — violates encapsulation or single-responsibility

---

## Expected issues to find (interviewer reference)

The candidate should identify at least 6 of these 10:

| # | Location | Issue | Classification |
|---|----------|-------|----------------|
| 1 | `OrderTrackerComponent` | Subscribes in `ngOnInit` but no `ngOnDestroy` — subscription leaks when component is destroyed | Memory leak |
| 2 | `OrderTrackingService` | `statuses` array is `public` — callers could mutate it externally, causing non-deterministic behaviour | API design |
| 3 | `OrderTrackingService` | `startTracking()` produces random (not sequential) statuses — calling it twice produces two concurrent intervals | Bug |
| 4 | `ReservationFormComponent` | `onSubmit()` uses `alert()` — blocks the main thread, fails in SSR, can't be tested | Missing error handling / API design |
| 5 | `ReservationFormComponent` | `timeElapsed` is logged every second in `ngOnInit` — console spam in production | Code quality |
| 6 | `AppComponent` | Navigation via boolean flags — URL doesn't update, browser history broken, no deep-linking | Coupling / architecture |
| 7 | `AppComponent` template | `*ngIf` eagerly creates all three child components — `OrderTrackerComponent` subscribes immediately | Memory leak / performance |
| 8 | `MenuComponent` | `menuItems` is hardcoded — no HTTP fetch, no loading state, no error state | Architecture |
| 9 | All forms | No global error handler — if any component throws, Angular silently logs to console | Missing error handling |
| 10 | `ReservationFormComponent` | `FormBuilder` is injected but `FormBuilder` import is not needed (can use `new FormGroup` or `inject(FormBuilder)` pattern inconsistently) | Code quality |
