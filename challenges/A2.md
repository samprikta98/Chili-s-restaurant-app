# A2 — State Management Architecture

**Level:** Architect
**Skills tested:** State design, signal-based store, selectors, NgRx trade-off analysis, derived state
**Files to create:**
- `src/store/app.store.ts`
- `src/store/app.store.spec.ts` (optional but encouraged)

---

## Context

Currently, state is scattered across the app:
- Cart state lives in `MenuComponent` (or `CartService` if M1 was done)
- Order tracking state lives in `OrderTrackingService`
- Reservation submissions are lost on navigation (no persistence)

This challenge asks you to design and implement a lightweight global state store using Angular Signals — no NgRx required — and justify when you would reach for NgRx instead.

### Task 2 — Migrate services to use the store
- Delete `CartService` (if it exists from M1) — move its logic into `AppStore`
- Update `OrderTrackingService` to call `appStore.setOrderStatus()` instead of updating its own `BehaviorSubject`/signal
- Update `ReservationFormComponent.onSubmit()` to call `appStore.addReservation()`

### Task 3 — Wire components to the store
- `MenuComponent`: inject `AppStore`, read `cart` and `cartTotal` signals, call `addToCart` / `removeFromCart`
- `OrderTrackerComponent`: read `orderStatus` and `hasActiveOrder` from the store
- `AppComponent` toolbar: show `cartItemCount()` badge from the store
- Reservation history page (or a section in `AppComponent`): list `reservations()` using `@for`

### Task 5 — NgRx trade-off discussion (no code)

Answer (discuss or write as comments in the store file):

1. This store has no action history. How would you implement **undo/redo** without NgRx?
2. NgRx DevTools provides **time-travel debugging**. When does this matter in production?
3. In a team of 10 engineers, what are the **collaboration trade-offs** of this custom store vs. NgRx?
4. When would you reach for **NgRx Component Store** vs. the global NgRx Store?
5. How would you add **optimistic updates** (e.g., add to cart immediately, roll back on API failure) in this signal store?

## Acceptance criteria

- [ ] `AppStore` has a clean read/write split: signals are private; public surface is readonly
- [ ] All derived values use `computed`, not stored signals
- [ ] `hasActiveOrder` drives the route guard from Task 4
- [ ] `cartTotal` and `cartItemCount` are correct after add/remove operations
- [ ] No component stores state locally that belongs in the store
- [ ] Candidate can clearly articulate when to use NgRx vs. a custom signal store
